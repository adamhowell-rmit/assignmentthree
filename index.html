    Group Project Idea "Goblin"

    Project Description (Concept)

    Goblin is our group project, it started out as a fundamentally stupid, clunky and useless idea called Slack’as. The idea of Slack’as was to somehow take the content of the forums RMIT uses, and put that on a slack server, the best way I figured “how” that could be done was to take the forum activity emails, forward them to a slack bot on a slack server, then post those forum posts to the relevant pre-made channels in the server. The main problem with this was that it relied on using the activity emails, and they are just not designed to be used like that. For example, they don’t include the entirety of longer posts, just the first few lines, they might have their format changed later down the line, throwing our bot into disarray. But the main glaring issue with Slack’as was sorting all the forum posts based of the limited information in the emails, the project was just too reliant on one source of information.

    Goblin was the idea the group arrived at for assessment 2, it was based off the same principle; taking text from one social media and putting it somewhere else in a usable format. But it was to be its own application, something that a user could copy paste their own text into, to be formatted and potentially pasted somewhere else. Goblin also had a more realistic approach what we could and couldn’t achieve at our skill level and was more aligned with the interests and capabilities of the group than previous versions.


    So why use an application like Goblin?

    Currently, it’s common to use a wide array of communication methods regularly. For example, right now as I type this, I have discord, slack, MS Teams, Gmail, and Outlook open on my PC. Now, in our daily life we are unlikely to find a need to record or move what goes on in one of those services to a different one, and on the off chance that you do, copy and pasting or screenshotting is usually good enough.

    However, in a hypothetical scenario where you are a student at RMIT and you are required to use the canvas forums, and later MS Teams, you may find yourself wishing that what was on the forums, or the unofficial discord was also on teams. This is where Goblin comes in! Goblin is aimed at making it easy to record and repost chat histories from different services.


    How will Goblin work?

    The application will consist of four major components: A searchable archive of all your chat histories if you want to find your response to a question on a forum that you no longer have access to, you can find it in your Goblin records.

    Making copying histories from one place to another while retaining a readable format easy. If you had a really informative private conversation about a project on discord, and would like to make it available to the group on MS Teams, Goblin could be used to format and post that entire conversation to teams.

    Reformatting text chat “dumps” Because Goblin reformats the pasted chat dump into a standard format, so that it can filter messages by date, sender, or keywords, it can provide data for analysis of chat records where the data is kept in its original form.

    Understanding data from unintegrated sources For example, if you wanted to know how many times the phrase “I hate MS Teams” was used on the unofficial Intro to Computer Science discord. Once you had stored the data in Goblin you would be able to know both the answer to that question and be able to see what messages contained the phrase. What’s unique about Goblin is that collecting this data doesn’t require an integration with the service that you’re taking the data from. No API calls, nothing.

    The ability to migrate histories of entire platforms from one place to another My personal project from ITT assignment 1 was a concept like Goblin, the idea was to forward the canvas forum update emails to a slack bot that would post the contents of the forum posts to a slack server, essentially automatically replicating the RMIT forums on slack. I received feedback that it was an interesting concept and could have potential use as an internal tool for a team testing the viability of moving the forums to slack. Goblin aims to be a more robust tool designed to assist in that sort of endeavor. It isn’t as sleek and fancy as something that is fully automatic and integrated, but it can be applied to a wide array of platforms with no additional programing.



    How will it work?

    The application can be thought of in 4 main parts

    Input:

    The input is the text that gets pasted in. this will then be sent to the middleware module to be processed

    Middleware:

    Once something has been entered, it needs to be broken up and stored. it’s expected that large amounts of text will be pasted in at once, so the program needs to break that text up into the original messages and store them. Data editing and deletion should be supported as well. When histories are stored there should be an option to add descriptions and tags, summaries as metadata.

    Output:

    When the database is searched it will output the desired chat.

    The UI:

    A user interface that combines the input and output features to make creation, search and deletion of chats easy and intuitive.


    As we’ll dive further into the description of this project a little later, we will provide screenshots on how the application is projected to operate.



    Technology:

    The key technologies that need to function for the application to accomplish what it should are as follows:


    Record management functionality:

    The app will need to do things like; save to an existing history, save as new history, delete history. As well as some filtering functionality that allows users to sort/locate individual messages or display sections of a history.

    This will be handled by SQLite a free SQL database manager that is easy to implement and widely used. SQLite was chosen as opposed to a larger package like MySQL because it’s lighter weight and simpler to use for a python program like Goblin. SQLite is also appealing because it runs on the local machine, instead of being cloud based like some other options I looked at. SQLite is also a relational database manager.

    A relational database is ideal for this use case because of how they’re intrinsically designed to associate one set of information (in this case date, source, subject matter, original sender as well as any other information that was attached when saved) with another (the content of the message). SQLite is a very commonly used tool, but it was chosen mainly for how easy it is to implement, and use was the main reason I think it should be used for database management.

    Managing the data:

    The point of Goblin is primarily a data manager for message histories. So, if nothing else the data management should be well implemented. SQLite will be what we use to handle this, and it was chosen specifically because it’s lightweight and easy to use.

    Having not been trained in SQLite at all, I spent about two or three hours talking to my dad about what needs to be done there to achieve what goblin needs. This is my preliminary understanding; there needs to be at least 2 tables, one that contains information attached to the actual snippet (an individual message) and another table for the tags that the user may specify about a given chat dump.

    The tag table needs to be separate because if it wasn’t it would massively complicate editing the tags if they were tangled up in the rest of the data about a snippet. The tags are the main way users can define information about what they’re storing. It’s expected that even with tags, what’s stored in goblin could get very messy. At this point in the design, it's up to the user to keep everything readable using the tagging method, so editing, deleting, and adding tags is a very important feature.

    The snippet table contains the inferred information like the author, date, and tag ID. As well as a user defined field called source, where the user defines where the snippet came from. The table also contains the snippet itself. The tag ID is what associates the tags in the tag table with the snippet table

    What other "goblins" are out there?

    How else is the problem of having messages in one place, and needing them in another solved?

    Well at the small-scale side, its handled by snip and sketch, you just “win+shift+s” and paste the relevant post wherever you want, nothing is more convenient or easier for an Indvidual to do. BUT it is problematic at scale, no one wants to catch up on a day’s worth of arguing over features in a google doc via screenshots.

    At the other extreme is the fully automatic approach that companies like Zendesk use internally. Some Zendesk customers like epic games for example use discord to handle customer interactions, but the customer service people don’t use discord, they use the Zendesk text window, that window sends 2 copies of the message out, one to discord, and the other to the internal location that the customer service interactions are stored. This works well for the relatively massive scale that Zendesk operates at, but such a solution is hilariously over engineered for a single user and would be impossible for even a single full-time developer to maintain.

    There are also SaaS companies like SimpleTexting or CallHub, which provide a way to personalize text message marketing to retailers. They gather and sell customer data to retailers to improve their SMS marketing. As part of that business, they manage databases of text messages.




    Detailed Description

    Aims

    Goblin aims to be a middle of the road between screenshotting and a completely automated api-webhook type solution. Goblin should be easy to use, and provide a way for single users, and possibly small teams to manage their text data across multiple platforms. Our vision for Goblin is that one day a user could take an entire chat history of a discord server from which they’re about to lose access, and with a single ctrl-v have all that information saved.

    Here are the main goals that need to be reached in order to produce the goblin we envision:

    A user interface; in order Goblin for goblin to be an app, it needs a user interface, but there's more to it than that. There are elements such as generating snippet ID’s that ideally would be created when the data is entered. Additionally, since all the database interaction takes place through the UI, how useful the database is will largely depend on how usable the UI is. The preliminary concept for the UI is outlined in detail further into the document, but the goal of the UI is to make it as intuitive, and as useful as possible.

    The backend infrastructure; the backend in this context is “all the stuff that doesn’t facilitate human interactivity”. This is required for things like the databases in which the snippets are stored, the searching algorithms, the code required to launch the application, windows API functions, and the like, there’s so much in this category that due to our inexperience we don’t even know we need but sorting that out definitely constitutes a goal.

    A message “sorter”; this is the most obviously fraught area of the project, frankly, having users paste large amounts of messages from an unknown source makes it a nightmare to isolate the Indvidual components. Even telling where one message ends and the next begins is extremely hard, because in the UI’s that these messages come from, they’re broken up into clearly separate messages by the UI, and the way it formats itself with profile pictures and such. As a group we haven't yet figured out any reliable way of sorting these messages without the use of a machine learning program, which we want to avoid, because a ML program will never be 100% accurate and any errors that it makes will be extremely obvious to users. So, sorting out the “sorting” issue is another main goal of the project.

    We think Goblin has the potential to be developed into a web-based application, rather than a local based application. This will allow the model of the project to really come to life, with the potential of a subscription-based modelling service to allow the application to be used across not just PC/MacOS, but also smart devices.

    Goblin also has the potential to become automated, with information/text not being required to be entered by a user, but by the application “sniffing” for the information. We imagine this could be done by “Application migration/connections”, an example of this would be how Spotify connects to Discord and broadcasts what music you are listening to at the time to your friends/server mates. Automation of information storage will be a huge convenience to the user.

    Design Log

    Database demonstration


    Initially the idea for the .db file scheme was to have two tables, one containing the snippet info, and the other containing the user given tags. In the snippet table, there would be a column for the author, date, message content, a unique message ID, and tag ID associated with that row.
